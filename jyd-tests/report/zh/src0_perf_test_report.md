# src0 性能测试报告

## 概述

`src0.bin` 包含一个名为 `run_perf_test_stage` 的第二阶段性能测试调度函数。
其行为如下：

- 在七段数码管上显示输入值
- 启动硬件计数器
- 运行三个带正确性交叉校验的工作负载
- 停止计数器
- 在七段数码管上显示耗时
- 在 LED 点阵上显示通过或失败

该函数在成功和失败路径上都返回 `0`；通过或失败状态通过 MMIO 副作用传达。

## 入口与显示行为

性能测试阶段开始时会调用：

- `show_input_on_seg()`
- `counter_start_pulse10()`

`show_input_on_seg()` 会读取位于 `0x80100000` 的 `g_dram_input`，使用 `pack_decimal_digits()` 将其转换为压缩十进制半字节表示，并将压缩后的数字写入 `MMIO_SEG` 的高字节通道。

`counter_start_pulse10()` 会向 `MMIO_COUNTER` 连续写入十次 `0x80000000`。
这强烈表明硬件计时器是沿触发或脉冲触发的，因此固件通过重复发出命令来提高鲁棒性。

## 工作负载 1：矩阵乘法交叉校验

第一轮循环执行 10 次，`matrix_iter = 0..9`。
每次迭代中：

- `fill_indexed_90x90(0x801138c4, matrix_iter)` 初始化矩阵 A
- `fill_indexed_90x90(0x8011b754, matrix_iter << 1)` 初始化矩阵 B
- `matrix_mul_90x90(0x801138c4, 0x8011b754, 0x801235e4)` 计算第一份结果
- `matrix_mul_reference_or_relayout(0x801138c4, 0x8011b754, 0x8012b474)` 计算第二份结果
- `matrix_eq_90x90(0x801235e4, 0x8012b474)` 检查两份输出是否一致

`fill_indexed_90x90()` 按如下方式填充一个 `90 x 90` 矩阵：

- `buf[row * 90 + col] = ((row * col) + seed) mod 100`

`matrix_mul_90x90()` 是一个标准的 `90 x 90 x 90` 三重循环矩阵乘法实现。

位于 `0x578` 的第二个矩阵辅助函数当前命名为 `matrix_mul_reference_or_relayout`，其体积较大，很可能充当参考实现、另一种数据布局实现，或基于重排的校验路径。
可以确定的是，它的作用是生成第二份矩阵结果以供比较。

如果 10 次矩阵校验中任意一次失败，样本会立即跳转到失败上报路径。

## 工作负载 2：排序交叉校验

第二轮循环同样执行 10 次，`sort_iter = 0..9`。
每次迭代中：

- `fill_linear_u32(0x80133304, 1000, sort_iter)` 填充一个 1000 元素的输入缓冲区
- 该缓冲区随后逐元素复制到 `0x801342a4`
- `bubble_sort_u32(0x80133304, 1000)` 对第一份副本进行排序
- `selection_sort_u32(0x801342a4, 1000)` 对第二份副本进行排序
- `array_eq_u32(0x80133304, 0x801342a4, 1000)` 比较排序后的输出

`fill_linear_u32()` 对每个元素按如下公式赋值：

- `(count * 0x49 - seed * 0x1f + index) mod 0x3037`

因此，这一阶段并不是在对两组不同数据进行基准测试，而是在同一组生成数组上对两种不同排序实现进行测试。

如果任意一次比较失败，样本会立即跳转到失败上报路径。

## 工作负载 3：素数计数交叉校验

在矩阵和排序阶段均通过后，样本会调用 `prime_test_stage()`。

该函数会对上限为 `0x4e20`（十进制 `20000`）的范围执行两种相互独立的素数计数方法，然后比较两者的计数结果：

1. 类试除法计数
   - 遍历 `n = 2..20000`
   - 调用 `is_prime_like(n)`
   - 若为素数，则递增 `trial_prime_count`

2. 类筛法计数
   - 将一个位于 DRAM 中的标记数组初始化为 `1`
   - 通过 `_DAT_80100038 = 0` 显式清除一个靠前的槽位
   - 遍历 `prime = 2..20000`
   - 若当前项非零，则将其计为素数
   - 从 `prime << 1` 开始，将其所有倍数标记为合数

只有在以下条件满足时，`prime_test_stage()` 才返回真：

- `sieve_prime_count == trial_prime_count`

因此，这一阶段同样是两个实现之间的正确性交叉校验，而不是纯吞吐导向的微基准测试。

## 通过/失败上报

在失败路径上，样本会调用 `report_perf_fail()`：

- `counter_stop_pulse10()`
- `read_counter()`
- `or_decimal_digits_to_seg(value)`
- `build_led_fail_pattern()`

在成功路径上，样本会调用 `report_perf_pass()`：

- `counter_stop_pulse10()`
- `read_counter()`
- `or_decimal_digits_to_seg(value)`
- `build_led_pass_pattern()`

这意味着 `MMIO_SEG` 的低 24 位被用来追加显示测得的耗时，而 LED 输出则以可视方式编码通过或失败状态。

## 结论

`src0` 性能测试最准确的描述是一个带正确性保护的基准测试阶段，包含三个工作负载：

- 矩阵乘法校验
- 排序校验
- 素数计数校验

它并不信任单一实现。
每个工作负载都会通过两种不同方法计算同一逻辑结果，并在首次出现不匹配时立即中止。
只有在所有校验均通过后，它才会停止计时器、显示耗时，并输出表示通过的 LED 图案。
